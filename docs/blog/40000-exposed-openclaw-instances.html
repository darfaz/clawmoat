Report #1: SecurityScorecard Finds 40K+ Exposed Instances

SecurityScorecard reported finding over 40,000 misconfigured OpenClaw instances exposed to the public internet, associated with 28,663 unique IP addresses.

The numbers are alarming:

  • 549 instances already correlated with prior breach activity
  • 1,493 instances associated with known vulnerabilities
  • 12,812 instances exploitable via remote code execution
  • 63% of all observed deployments are vulnerable

Most exposures are in China, followed by the US and Singapore. Information services is the most impacted industry.

"The more centralized the access, the more damage a single compromise can cause. What looks like convenience is actually a concentration of risk." — SecurityScorecard

And it gets worse: threat actors are already targeting agents with infostealers.

As Endor Labs put it:

"The multi-layer architecture of AI agent frameworks means vulnerabilities often span multiple files and components. Understanding the complete source-to-sink path is critical."

Why Sandboxes Alone Don't Fix This

The instinctive response to these reports is "just run it in a sandbox." And sandboxes help — they contain blast radius. But they miss critical attack vectors:

  • Credential access: Your agent needs credentials to be useful. A sandbox doesn't prevent the agent from reading ~/.ssh/id_rsa or ~/.aws/credentials within its own scope.
  • Prompt injection: Malicious instructions embedded in emails, websites, or messages execute within whatever permissions the agent has — sandbox or not.
  • Malicious skills: Skills installed from ClawHub run as trusted code. A sandbox doesn't distinguish between a legitimate skill and one that exfiltrates your API keys.
  • Network egress: The agent needs network access. A sandbox doesn't monitor what data leaves through allowed channels.

As one Hacker News commenter noted: "I don't think OpenClaw can possibly be secured given the current paradigm. It has access to your personal stuff (that's its main use case), access to the net, and it gets untrusted third party inputs. That's the unfixable trifecta."

They're partially right. You can't eliminate the risk. But you can monitor, detect, and limit it at the host level.

The Missing Layer: Host-Level Runtime Protection

SecurityScorecard's own recommendations point to what's needed:

  1. Aggressively limit access — grant only what's needed
  2. Adopt zero trust — never trust, always verify
  3. Monitor the logic, instructions, and components
  4. Treat every agent like a privileged identity

This is exactly what host-level protection does. Not instead of sandboxes — alongside them.

What Host Protection Catches That Sandboxes Don't

  • Agent reading credential files outside its working directory
  • Skills with obfuscated code or suspicious network calls
  • Permission escalation beyond the assigned tier
  • Data exfiltration through allowed network channels
  • Behavioral anomalies (3 AM file access, unusual command patterns)

Practical Steps You Can Take Today

1. Check if you're exposed

If your OpenClaw instance is accessible from the internet, you're part of that 40,000. Check your firewall rules. OpenClaw should never be exposed to the public internet.

2. Update immediately

All six Endor Labs vulnerabilities have patches. Run npm update -g openclaw or update your Docker image.

3. Audit your skills

Review every installed skill. Remove anything you're not actively using. Check skill source code for suspicious patterns.

4. Add runtime monitoring

npm install clawmoat

# Scan a skill before installing
npx clawmoat skill-audit ./path-to-skill

# Run with host protection
npx clawmoat --tier worker --audit-log ./agent-audit.json

ClawMoat adds the host protection layer: permission tiers, forbidden zone enforcement, credential monitoring, skill integrity checking, and network egress logging. Open source, zero dependencies, 142 tests.

5. Don't run on your primary workstation

Microsoft's advice is still sound: use a dedicated machine or VM. But if you must run on your workstation (most people do), at minimum enforce permission tiers and monitor file access.

The Bigger Picture

These reports confirm what many of us have been saying: the OpenClaw ecosystem grew faster than its security model. The OpenClaw Foundation (under OpenAI since February 15) has been patching vulnerabilities, but the fundamental architecture — an agent with broad system access processing untrusted inputs — requires defense in depth.

No single tool fixes this. You need:

  • Sandboxing for blast radius containment
  • Host monitoring for runtime behavior detection
  • Skill auditing for supply chain security
  • Network controls for egress filtering
  • Human oversight for approval of sensitive operations

ClawMoat handles three of those five layers. It's not a silver bullet — nothing is. But it's the layer most deployments are missing entirely.
